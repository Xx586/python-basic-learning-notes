# 04 — 按顺序完善多个章节

> **用途**：按章节顺序完善用户指定范围的多个章节。每章完成后自检，再继续下一章。
>
> **依据**：`references/workflow.md` 第 3-4 步及"分批处理原则"，Session 3 第 6-10 章完善 prompt

---

## 请执行以下任务

请按照当前项目文件夹中已经生成好的知识点框架，继续完善 **第 X 章到第 Y 章** 的知识点讲解内容。

本轮任务范围只包括第 X 章到第 Y 章，请不要处理其他章节。

---

## 一、开始前必须先读取的资料

请先读取当前项目文件夹中的以下内容：

1. `CLAUDE.md`，如果存在。
2. 当前文件夹中保存的最详细 prompt 文件。
3. `00_资料总框架.md`，如果存在。
4. 资料盘点报告，如果存在。
5. 学习路线文件。
6. 学习规范文件。
7. `思维导图.png`。
8. 所有截图中的知识点结构。
9. 当前第 X-Y 章目录下的所有 `.md` 文件。
10. 第 X-Y 章原有旧资料、已有代码文件。

---

## 二、执行方式

请按章节顺序逐章处理：

1. 先处理第 X 章
2. 再处理第 X+1 章
3. ...直到第 Y 章

**每完成一个章节后，必须进行本章自检，再继续下一个章节。**

---

## 三、每章处理流程

1. 读取本章所有已有 `.md` 文件
2. 对每个知识点文件，判断处理方式：

| 文件状态 | 处理方式 |
|---------|---------|
| 已有完整内容 | 保持原样，不做修改 |
| 空模板 / 待完善 | 完全替换占位内容，从零撰写 |
| 内容不完整 | 在原有内容基础上追加缺失部分 |
| 超大文件需拆分 | 拆分为多个独立文件，内容完整保留 |

3. 按 8 部分框架完善 / 补齐内容（参考 `references/knowledge_point_rules.md`）
4. 生成或更新 `demo/README.md`
5. 执行本章自检（13 项检查标准）
6. 自检通过后进入下一章

---

## 四、每章自检清单

每章完成后逐项检查：

1. [ ] 本章所有知识点文件是否都已完善（非空、非"待完善"）
2. [ ] 每个知识点是否都有完整的 8 个部分
3. [ ] 每个知识点是否都有 Demo 样例
4. [ ] Demo 代码是否可运行（非伪代码）
5. [ ] Demo 是否有注释（在代码旁边或上方）
6. [ ] Demo 是否有运行结果
7. [ ] 是否有典型例题（1-2 道）
8. [ ] 典型例题是否附带答案和解析
9. [ ] 是否有课后练习（2-3 道）
10. [ ] 课后练习是否附带答案（代码可运行）
11. [ ] 是否存在内容空泛、重复、过短的问题
12. [ ] 是否适合零基础学生学习
13. [ ] 是否有遗漏学习路线、思维导图、截图中的知识点

**发现问题请立即修复当前章节，不带着已知问题进入下一章。**

---

## 五、分批处理原则

- 按章节序号顺序处理，不跳章。
- 复杂章节（第 4、5、10 章）需要更多时间，不要匆忙。
- 如果上下文或任务过长，请优先保证当前章节质量，不要为了赶进度压缩内容。
- 如果 token 不足或上下文超长，请在当前章节完成后暂停，等待用户说"继续"。
- 如果发现某章工作量过大，可以在完成该章后主动暂停。

---

## 六、输出

全部指定章节完成后，请输出完成报告，格式如下：

```markdown
# 第 X-Y 章完善完成报告

## 1. 已处理章节

| 章节 | 知识点文件数量 | 是否全部包含 8 个部分 | Demo 是否完整 | 例题是否完整 | 练习是否完整 |
|---|---:|---|---|---|---|

## 2. 每章知识点覆盖情况

（每章分别说明已覆盖的知识点、补充的知识点、需确认的知识点）

## 3. Demo 检查情况

## 4. 典型例题与课后练习情况

## 5. 原始资料处理情况

## 6. 后续建议
```

---

## 安全限制

1. 只处理第 X-Y 章，不处理其他章节。
2. 不删除原始资料，不覆盖重要文件。
3. 如果要修改已有文件，请在原文件基础上补充完善。
4. 如果发现同名文件或冲突文件，先说明再处理。
5. 如果需要新增知识点文件，必须在报告中说明。
6. 不要把所有内容写到一个大文件里。
7. 每个知识点必须保持独立 `.md` 文件。
8. 任务过长时暂停，等待用户继续。
