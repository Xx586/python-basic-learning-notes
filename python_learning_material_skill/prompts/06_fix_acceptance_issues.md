# 06 — 修复验收问题

> **用途**：根据验收报告中的 A/B/C 问题清单，按优先级逐类修复。
>
> **依据**：`references/acceptance_rules.md` 第 5-8 节，`references/workflow.md` 第 6 步，`templates/issue_fix_report_template.md`

---

## 请执行以下任务

请根据最终验收报告中的问题清单，按照 A → B → C 的优先级顺序修复问题。

除非我明确说"开始修复"，否则不要改动任何章节内容。

---

## 执行流程

### 第 1 步：确认修复范围

请先确认你要修复的问题类别和范围。

用户可以指定：
- "修复全部 A 类问题"
- "只修复 A 类问题"
- "修复 A + B 类问题"
- "修复编号 A1-A5、B1-B3"
- 或直接指定修复某个具体文件

### 第 2 步：按优先级逐类修复

修复优先级必须是：
1. **先修复 A 类问题**（必须修复）：
   - 文件缺失 → 创建对应文件
   - 缺少 8 部分结构 → 补全缺失部分
   - 大量"待完善" → 替换为完整内容
   - Demo 不可运行 → 修复或重写 Demo
   - 典型例题或练习缺答案 → 补充答案
   - 明显知识点遗漏 → 新建知识点文件
   - 明显概念错误 → 修正错误内容

2. **再修复 B 类问题**（建议修复）：
   - 内容太短 → 补充讲解
   - 注释不够清晰 → 重写注释
   - 易错点不够具体 → 补充错误代码和正确代码
   - 例题不够贴近 → 更换例题
   - 练习层次不明显 → 调整练习难度顺序

3. **最后修复 C 类问题**（可后续优化）：
   - 增加图示、更多练习、项目案例等

### 第 3 步：输出修复报告

每类问题修复完成后，请输出修复报告。

报告格式：


## 修复规则

### A 类修复规则

1. **文件缺失**：根据总框架中的知识点清单，创建对应的 `.md` 文件，使用 8 部分框架从零撰写。
2. **缺少 8 部分结构**：在已有文件中补齐缺失的部分标题和内容。如果原文件有可用内容，保留并重组为 8 部分格式。
3. **"待完善"占位**：将"待完善"内容替换为完整讲解。
4. **Demo 不可运行**：检查代码语法错误，修复或重写 Demo。
5. **缺答案**：为缺答案的例题和练习补充完整答案和解析。
6. **知识点遗漏**：新建对应的知识点 `.md` 文件。
7. **概念错误**：修正错误内容，确保概念正确。

### B 类修复规则

1. **内容太短**：补充更详细的讲解、更多例子。
2. **注释不够清晰**：重写注释，解释"做什么"和"为什么"。
3. **易错点不够具体**：补充错误代码示例和正确写法。
4. **例题不够贴近**：更换为更贴合知识点的例题。
5. **练习层次不明显**：调整练习顺序，确保由易到难。

### C 类修复规则

1. 仅做增量补充，不做大规模重构。
2. 增加图示、更多练习等以不破坏现有结构为前提。

---

## 安全限制

1. 按 A → B → C 优先级修复，不跨级。
2. 每类修复后输出报告，让用户确认。
3. 不重构整个目录。
4. 不修改与当前问题无关的章节。
5. 不删除任何原始资料。
6. 如果需要修改已有文件，先在原文件基础上补充完善。
7. 如果发现新的问题，在报告中说明但不擅自扩大修复范围。
8. 修复后务必重新检查相关问题是否已解决。
