# 完整工作流定义

> **来源**：从 `CLAUDE.md`、`promopt.txt` 和 5 次 Claude Code 对话 JSONL transcript 中提炼。
> 
> **适用场景**：无论从零开始还是增量完善，以下 8 步流程覆盖了从资料盘点到最终交付的完整生命周期。

---

## Skill 来源说明

本 Skill 的工作流程从以下真实执行记录中提炼：

1. **Session 1**（资料盘点）：读取所有原始资料 → 输出盘点报告
2. **Session 2**（前五章完善）：基于已有框架 → 逐章完善第 1-5 章
3. **Session 3**（第 6-10 章完善）：继续完善第 6-10 章
4. **Session 4**（最终验收 + 修复）：10 项验收检查 → A/B/C 分类 → A1-A3 修复
5. **Session 5**（Skill 准备）：查找资料 → 提炼设计方案 → 生成 Skill

---

## 第 0 步：准备工作

### 目标
建立执行上下文，确保所有后续决策都有依据。必须先读完资料，再开始任何生成工作。

### 输入
- 用户指定的目标文件夹路径
- 用户指定的任务范围（全部 10 章 / 部分章节 / 仅验收）

### 必须读取的资料（按优先级排序）

| 序号 | 文件 | 作用 | 必需程度 |
|:--:|------|------|:--:|
| 1 | `CLAUDE.md` | 项目规则，最先阅读，建立基本约束 | **必需** |
| 2 | 最详细的 prompt 文件（如 `promopt.txt`） | 完整生成规范，验收标准依据 | **必需** |
| 3 | 学习路线文件（如 `Python基础学习路线(5).md`） | 知识点主线依据 | **必需** |
| 4 | 学习规范文件（如 `2.AI应用学习方式_副本.md`） | 文件组织规范 | 强烈建议 |
| 5 | `思维导图.png` | 知识点结构全景 | 强烈建议 |
| 6 | 所有截图（多张 .png） | 知识点结构补充验证 | 建议 |
| 7 | 现有 1-10 章节目录下的 `.md` 文件 | 已有内容参考 | 建议 |
| 8 | 现有 `.py` Demo 文件 | Demo 参考 | 建议 |
| 9 | JSONL transcript（如有） | 历史对话记录 | 可选 |

### 执行规则
- 如果截图内容不清晰，结合文件名、目录结构和学习路线推断，但在报告中标注"不确定"。
- 如果学习路线、思维导图、截图、旧资料之间存在不一致，**以学习路线为主**，思维导图和截图作为补充，旧资料作为参考。
- 如果某些资料缺失，在报告中说明，但不阻止流程推进。

---

## 第 1 步：资料盘点

### 目标
输出完整的资料分析结果。**本轮只输出报告，不修改任何文件。**

### 输入
- 第 0 步已读取的所有资料

### 执行动作
1. 扫描目标文件夹中的所有内容（按类型分组列出）
2. 阅读所有现有资料（包括 .md、图片、截图、章节文件夹、代码文件）
3. 输出资料盘点报告

### 必须输出的内容
1. 已识别到的文件和资料（按类型分组：.md / .py / .png / .txt / 其他）
2. 已识别到的章节结构（现有目录 vs 学习路线对比）
3. 已识别到的核心知识点（按章节分组）
4. 现有资料中可以保留的内容
5. 现有资料中需要补充的内容
6. 现有资料中可能重复或需要合并的内容
7. 需要新增的知识点文档
8. 整理风险说明（不确定的内容要明确标注）

### 安全规则
- 本轮只盘点，不修改任何文件。
- 不生成任何理论文档。
- 不确定的内容标注"不确定"，不要编造。

---

## 第 2 步：生成知识点总框架

### 目标
生成 `00_资料总框架.md` 并等待用户确认。

### 输入
- 第 1 步的资料盘点报告
- 学习路线、思维导图、截图

### 执行动作
1. 输出本章知识点清单（每个章节有哪些知识点）
2. 输出本章整理方案（哪些保留 / 哪些补充 / 哪些合并 / 哪些重写 / 哪些新建）
3. **等待用户确认**
4. 创建或更新 `00_资料总框架.md`

### 框架文件必须包含
- **10 个章节总览表**：章节名称、知识点文件数量、学习路线阶段、难度、说明
- **每个章节的知识点清单表**：序号、知识点名、文件名、来源标注、状态标注
- **跨章节知识点说明**：如"推导式在第 3 章也有覆盖"
- **现有资料处理说明**：保留不变 / 新建 / 保留参考 / 待移除
- **后续完善顺序建议**：按章节序号列出优先顺序
- **每章完善时的执行要求**：补全 8 个部分的具体要求
- **文件命名规范**：序号.知识点中文名.md，特殊字符简化规则

### 来源标注格式
```
| 2.1 | 基本数据类型 | `1.基本数据类型.md` | 学习路线⑥ + promopt + 已有旧资料 | ✅ 已有内容 |
| 2.6 | 表达式 | `6.表达式.md` | promopt 明确要求 | 🆕 待完善 |
```

### 安全规则
- 等待用户确认后再生成框架。
- 不展开任何理论正文。
- 不删除或覆盖已有文件。

---

## 第 3 步：分章节完善

### 目标
按章节顺序，逐章完善每个知识点 `.md` 文件。

### 核心原则
- **每次只处理一个章节**，不要一次性生成全部 10 章。
- 在现有框架基础上补充，不要脱离已有框架重新生成。
- 如果已有 `.md` 文件中已有标题框架，直接在对应标题下补充内容。

### 每章内部处理流程
1. 读取本章所有已有 `.md` 文件
2. 对每个知识点文件，判断处理方式（见"文件区分处理方式"）
3. 按 8 部分框架完善 / 补齐内容（见 `knowledge_point_rules.md`）
4. 生成或更新 `demo/README.md`
5. 执行本章自检（见第 4 步）
6. 自检通过后进入下一章

### 文件区分处理方式

| 文件状态 | 处理方式 | 说明 |
|---------|---------|------|
| **已有完整内容** | 保持原样，不做修改 | 质量已达标，无需改动 |
| **空模板 / 待完善** | 完全替换占位内容，从零撰写 | 以完整 8 部分框架填充 |
| **内容不完整** | 在原有内容基础上追加缺失部分 | 保留有价值的旧内容，补充典型例题、易错点、课后练习、答题总结 |
| **超大文件需拆分** | 拆分为多个独立文件 | 内容完整保留，按主题拆分为独立的 .md 文件 |

### 输出
- 完整的知识点 `.md` 文件（每章 N 个）
- 每章 `demo/README.md`（含学习顺序、Demo 与理论对应关系、运行方式、学习建议）

---

## 第 4 步：章节自检

### 目标
每章完成后立即自检，不带着已知问题进入下一章。

### 13 项检查标准

| 序号 | 检查项 | 标准 |
|:--:|------|------|
| 1 | 知识点文件完整性 | 本章所有知识点文件是否都已完善（非空、非"待完善"） |
| 2 | 8 部分结构 | 每个知识点是否都有完整的 8 个部分 |
| 3 | Demo 存在 | 每个知识点是否都有 Demo 样例 |
| 4 | Demo 可运行 | Demo 代码是否可运行（非伪代码） |
| 5 | Demo 注释 | Demo 是否有注释（在代码旁边或上方，解释"做什么"和"为什么"） |
| 6 | Demo 运行结果 | Demo 后是否写了预期运行结果 |
| 7 | 典型例题 | 是否有 1-2 道典型例题 |
| 8 | 例题答案 | 典型例题是否附带答案和解析 |
| 9 | 课后练习 | 是否有 2-3 道课后练习 |
| 10 | 练习答案 | 课后练习是否附带答案（代码可运行） |
| 11 | 内容质量 | 是否内容空泛、重复、过短 |
| 12 | 适合零基础 | 是否适合零基础学生理解（语言通俗、有例子、有类比） |
| 13 | 知识点覆盖 | 是否遗漏学习路线、思维导图、截图中的知识点 |

### 发现问题的处理方式
- **立即修复**当前章节，再进入下一章节。
- 不在报告中标注"已知问题但未修复"。

---

## 第 5 步：最终验收

### 目标
对 10 章整体执行全面验收，输出 A/B/C 三级问题清单。**本轮只验收，不修改。**

### 10 项验收检查

| 序号 | 检查项 | 检查内容概要 |
|:--:|------|------|
| 1 | 章节完整性 | 10 个章节目录是否都存在，命名是否一致 |
| 2 | 知识点文件完整性 | 全部文件是否齐备，无空文件，无"待完善"占位 |
| 3 | 8 部分结构 | 每个文件是否完整包含 8 个部分，标题顺序是否正确 |
| 4 | Demo 样例 | 15 项子检查（可运行 / 有注释 / 有运行结果 / 适合零基础等） |
| 5 | 典型例题 | 是否有答案和解析，难度是否适合零基础 |
| 6 | 课后练习 | 是否有答案，是否由易到难，是否超纲 |
| 7 | 易错点 | 是否有错误写法 + 原因 + 正确写法 |
| 8 | 内容质量 | 是否适合零基础、通俗易懂、无空泛重复 |
| 9 | 学习路线覆盖 | 44 个核心知识点是否全部覆盖 |
| 10 | 思维导图和截图覆盖 | 图片中的知识点是否遗漏 |

### 输出
- 最终验收报告（使用 `templates/final_acceptance_report_template.md`）
- A/B/C 三级问题清单
- 修复优先级建议

### 安全规则
- 本轮只验收，不直接大规模修改。
- 不重新生成任何章节。
- 不删除任何原始资料。
- 不确定的内容标注"不确定"，不要编造。

---

## 第 6 步：问题修复

### 目标
根据验收报告中的问题清单，按优先级逐类修复。

### 修复优先级
1. **先修复 A 类问题**（必须修复）：结构缺失、文件缺失、Demo 不可运行、缺答案
2. **再修复 B 类问题**（建议修复）：内容太短、注释不清晰、易错点不够具体
3. **最后修复 C 类问题**（可后续优化）：命名优化、增加图示、更多练习

### 执行规则
- 用户可以指定只修复某一类（如"只修复 A 类"）。
- 每类修复完成后输出修复报告（使用 `templates/issue_fix_report_template.md`）。
- 修复后重新检查相关问题是否已解决。

### 输出
- 修复报告（含修复前后的状态对比和验证结果）

---

## 第 7 步：生成 / 更新总 README

### 目标
生成或更新项目根目录的 `README.md`。

### 必须包含的 11 项内容
1. 学习目标
2. 适合人群
3. 10 个章节的学习顺序（含预计时间和难度）
4. 每章主要学习内容
5. 每个知识点文档的固定结构说明
6. Demo 样例的学习方法
7. 如何运行 Python 代码（命令行 / IDE / 交互式）
8. 零基础学习建议（7 条）
9. 常见报错处理方式（含速查表）
10. 如何更新学习进度（checkbox 清单）
11. 后续可以继续学习的方向（含前置要求）

---

## 第 8 步：沉淀和复用

### 目标
将本次执行的经验沉淀为可复用资产。

### 可选动作
- 将关键决策和发现写入 Claude Code memory
- 如有 JSONL transcript，提取关键 prompt 模板
- 可选：生成或更新 Skill 文件

---

## 分批处理原则

| 序号 | 原则 | 原因 |
|:--:|------|------|
| 1 | 不要一次性生成全部 10 章 | 上下文过长导致质量下降 |
| 2 | 每次只处理一个章节 | 保证每章质量 |
| 3 | 先资料盘点，再生成总框架 | 建立全局视图后再动手 |
| 4 | 每章完成后自检 | 不带着问题进入下一章 |
| 5 | 优先处理有待完善文件的章节 | 消除"占位"内容 |
| 6 | 复杂章节（第 4、5、10 章）更需集中处理 | 内容多、难度高 |
| 7 | 按章节序号顺序处理 | 1→2→3→...→10，保持逻辑连贯 |
| 8 | 任务过长时暂停，等待用户继续 | 不强行压缩内容 |
| 9 | 用户可指定只处理特定章节范围 | 灵活支持增量完善 |

## 章节难度参考

| 难度 | 章节 | 处理建议 |
|:--:|------|------|
| ⭐ | 1, 2, 7 | 基础内容，可适当加快 |
| ⭐⭐ | 3, 6, 8 | 中等难度，正常处理 |
| ⭐⭐⭐ | 4, 9 | 较难内容，需要细心 |
| ⭐⭐⭐⭐ | 5, 10 | 最难内容，需要最多时间 |
