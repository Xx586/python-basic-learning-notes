# 知识点文档规则集

> **来源**：从 `promopt.txt` 第五节至第九节、`CLAUDE.md` 第三节至第四节提炼。
>
> **用途**：生成每个知识点 `.md` 文件时必须对照的"规范字典"。

---

## 1. 固定 8 部分结构

每个知识点 `.md` 文件必须严格按照以下 8 部分结构，标题名称和顺序不可变更：

```markdown
# 知识点名称

## 1. 知识点定义

## 2. 语法格式

## 3. 重点说明

## 4. Demo 样例

## 5. 典型例题（附带答案）

## 6. 易错点

## 7. 课后练习（附带答案）

## 8. 答题总结
```

**结构规则**：
- 不允许缺失任何部分。
- 不允许将"待完善""TODO""后续补充"等占位内容作为最终产出。
- 每个部分必须有实质内容，不能只有空标题。
- 标题名称和顺序必须与上述一致。

---

## 2. 第 1 部分：知识点定义

### 核心要求
- 面向 **Python 零基础学生**。
- 用通俗语言解释，不要照搬官方文档。
- 不要写得过于抽象。
- 可以使用生活类比。
- 要说明该知识点是什么。
- 要说明它解决什么问题。
- 要说明为什么需要学它。

### 建议格式
```markdown
这个知识点可以理解为……

它主要解决的问题是……

在实际写 Python 代码时，它常用于……
```

### 可选增强（来自 promopt.txt）
```markdown
> 小白理解：
> 这里可以把变量理解成一个带名字的盒子，盒子里可以放数字、文字、列表等数据。
```

---

## 3. 第 2 部分：语法格式

### 核心要求
- 写出最常用、最基础、最实用的语法格式。
- 使用 Python 代码块展示。
- 每种语法后面必须有解释。
- 语法必须准确，不能是伪代码。
- 不要一上来就堆高级写法。

### 示例
````markdown
```python
变量名 = 值
```

说明：变量名用来保存数据，等号右边是要保存的具体内容。
````

---

## 4. 第 3 部分：重点说明

### 核心要求
- 结合原始资料中的重点内容。
- 结合思维导图和截图中的知识结构。
- 结合零基础学习常见问题。
- **讲清楚"什么时候用"**。
- **讲清楚"和相似知识点有什么区别"**。
- **讲清楚"初学者最容易混淆的地方"**。
- 可以使用表格进行对比。

### 可选格式
```markdown
> 小白理解：
> ……

| 对比项 | A | B |
|---|---|---|
| 使用场景 | ... | ... |
| 关键区别 | ... | ... |
```

## 6. 第 5 部分：典型例题

### 数量要求
- 每个知识点 1-2 道。

### 内容要求
- 题目必须围绕当前知识点。
- 难度适合零基础学生。
- 必须附带完整答案。
- 答案必须有必要解析。
- 不能只给代码不解释。
- 例题要贴近课堂作业、考试或常见编程问法。

### 格式
```markdown
### 例题 1

**题目：**

**答案：**

**解析：**
```

## 8. 第 7 部分：课后练习

### 数量要求
- 每个知识点 2-3 道。

### 内容要求
- 练习难度由易到难。
- 练习必须巩固当前知识点。
- 每道练习都必须附带答案。
- 答案代码必须可运行。
- 不要设计超出当前知识点太多的题目。
- 可以让学生修改 Demo 中的变量、条件、数据、函数或类。

### 格式
````markdown
### 练习 1

**题目：**

**答案：**

```python
答案代码
```

**说明：**
````

## 9. 第 8 部分：答题总结

### 核心要求
- 用 3-5 条 bullet 总结。
- 适合写进笔记里复习。
- 适合作业或考试前快速回顾。
- 总结该知识点的定义、语法、使用场景、易错点。
- 语言要简洁清晰。

### 示例
```markdown
- 变量用于保存程序运行过程中需要使用的数据。
- Python 中定义变量不需要提前声明类型。
- 变量名要见名知意，不能使用 Python 关键字。
- 初学者要注意变量未定义、拼写错误和中文符号问题。
```

## 10. Demo 生成规则

### 基础要求（15 条）

| 序号 | 规则 | 说明 |
|:--:|------|------|
| 1 | 服务于当前知识点 | 不能随便放一段无关代码帮助理解理论知识 |
| 2 | 真实可运行 | 不能是伪代码 |
| 3 | 尽量用标准库 | 不依赖第三方库，除非当前知识点讲 pip、模块、包或第三方库 |
| 4 | 场景贴近学习/生活 | 学生成绩、购物计算、温度转换、名片管理等 |
| 5 | 有清晰注释 | 注释写在代码对应行旁边或上方 |
| 6 | 注释解释"做什么"和"为什么" | 关键变量、判断、循环、函数调用、类定义、文件操作、异常处理必须解释 |
| 7 | 不写无意义注释 | 如 `a = 1  # a 等于 1` 是无意义的 |
| 8 | 不单独新建"代码讲解文件" | 注释直接写在代码旁边 |
| 9 | Demo 后写预期运行结果 | 使用 `运行结果：` 代码块 |
| 10 | 不重复已有 Demo | 可以基于旧代码优化 |
| 11 | 文件操作类说明文件影响 | 说明会生成或读取哪些文件 |
| 12 | 异常处理类体现捕获效果 | 能清楚展示异常被捕获后的效果 |
| 13 | 正则类展示匹配结果 | 显示匹配 / 替换前后的对比 |
| 14 | 迭代器/生成器类体现惰性计算 | 展示"逐个生成""节省内存"的思想 |
| 15 | 面向对象类体现类和对象关系 | 清晰体现类、对象、属性或方法 |

### 代码注释正确示例
```python
age = 18  # 用 age 保存年龄，后面可以根据年龄判断是否成年
```

### 代码注释错误示例（禁止）
```python
a = 1  # a 等于 1
```

### 运行结果格式
````markdown
```text
运行结果：
及格
```
````

## 11. 易错点规则

### 格式要求
````markdown
### 易错点 1：xxx

**错误写法：**

```python
错误代码
```

**错误原因：**

**正确写法：**

```python
正确代码
```
````

### 内容要求
- 每个易错点都要包含：错误写法 + 错误原因 + 正确写法。
- 尽量结合 Python 常见报错信息。
- 每个知识点至少 2-3 个易错点。

### 按章节列出的重点易错领域

| 章节范围 | 重点易错领域 |
|---------|------------|
| 第 1-3 章 | 缩进错误、中文符号（中英文标点混淆）、变量未定义（NameError）、类型错误（TypeError）、赋值（=）与比较（==）混淆、Tab 和空格混用 |
| 第 4-5 章 | 默认参数使用可变对象（list/dict）、作用域混淆（LEGB 规则不清）、self 忘记写、多继承 MRO 理解错误、`__` 名称改写误解 |
| 第 6 章 | 文件路径写错（相对路径和绝对路径混淆）、忘记关闭文件、编码错误（encoding 未指定）、JSON 字符串和 Python 字典混淆、捕获异常过于宽泛（`except:` 或 `except Exception:`） |
| 第 7 章 | import 路径错误、包缺少 `__init__.py`、循环导入、pip 安装位置不一致（全局 vs 虚拟环境）、`__name__ == "__main__"` 判断遗漏 |
| 第 8 章 | 正则表达式转义错误（忘记用原始字符串 `r"..."`）、贪婪匹配和非贪婪匹配混淆、match vs search 的区别不清、findall 在有无分组时返回值不同 |
| 第 9 章 | 迭代器只能遍历一次（遍历完变空）、生成器不会一次性返回所有结果、生成器表达式用 `()` 不是 `[]`、yield vs return 的区别 |
| 第 10 章 | 浅拷贝和深拷贝混淆（copy.copy vs copy.deepcopy）、async 函数忘记 await、with 语句原理不清（`__enter__`/`__exit__`）、解包时变量数量不匹配、多线程 vs 多进程的选择（GIL） |

## 12. 旧资料处理规则

以下 8 种情况对照表（来自 promopt.txt 第六节）：

| 情况 | 处理方式 | 原则 |
|------|---------|------|
| 原文正确但太简略 | 保留原文，在对应位置补充解释 | 不删除原有内容 |
| 原文有代码但缺少讲解 | 保留代码，并给代码加注释 | 不删除原有代码 |
| 原文适合有基础的人但不适合零基础 | 增加"小白理解版" | 降低理解门槛 |
| 原文内容重复 | 合并成一份更清晰的讲解 | 去重不丢失信息 |
| 原文有错误 | 修正错误，并避免保留错误说法 | 以学习路线为准 |
| 原文与学习路线冲突 | 以学习路线为准 | 学习路线优先级最高 |
| 学习路线有但旧资料没有 | 新建对应知识点 `.md` 文件 | 不遗漏知识点 |
| 截图或思维导图有但旧资料没有 | 作为补充知识点加入合适章节 | 不遗漏图片知识点 |

### 补充内容格式
```markdown
> 小白理解：
> 这里可以把变量理解成一个带名字的盒子，盒子里可以放数字、文字、列表等数据。

> 补充说明：
> 这个知识点在实际写代码时经常用于……
```

## 13. 内容质量要求

1. 内容必须适合 Python 零基础学生。
2. 语言要通俗，不要写成官方文档风格。
3. 不要只写概念，要结合例子讲。
4. 不要生成空泛、重复、模板化严重的内容。
5. 不要为了凑字数写废话。
6. 每个知识点都要真正可用于学习。
7. Demo 必须真实可运行。
8. Demo 代码必须有注释。
9. Demo 后必须写运行结果。
10. 典型例题必须附带答案和解析。
11. 课后练习必须附带答案。
12. 每个知识点内容要完整，但不要过度冗长。
13. 不能遗漏知识点。
14. 如果发现知识点缺失，先在报告中说明再创建。
15. 如果发现知识点归类明显不合理，先说明不要擅自大规模移动。

---

## 14. 文件命名规范

- 知识点文件：`序号.知识点中文名.md`
- 特殊字符简化（在正文中说明实际写法）：
  - `*args` / `**kwargs` → 文件名中用"可变参数"替代
  - `__init__` → 文件名中用"init构造方法"替代
  - `__name__ == "__main__"` → 文件名中用"name_main入口判断"替代
- 每章 `demo/` 目录：存放 `.py` 示例代码和 `README.md`

---

## 15. 章节文件结构规范

```
N.章节名称/
├── 1.知识点A.md          # 理论讲解（含 8 部分框架 + Demo）
├── 2.知识点B.md          # 理论讲解（含 8 部分框架 + Demo）
├── 3.知识点C.md          # 理论讲解（含 8 部分框架 + Demo）
└── demo/
    ├── README.md          # 学习顺序 + Demo说明 + 理论对应关系
    ├── 1_demo_a.py        # 知识点A的操作实例（可选）
    ├── 2_demo_b.py        # 知识点B的操作实例（可选）
    └── 3_demo_c.py        # 知识点C的操作实例（可选）
```

### demo/README.md 内容要求
1. 推荐学习顺序
2. Demo 与理论文档对应关系（表格）
3. 如何运行 Demo
4. 学习建议（5 条）
