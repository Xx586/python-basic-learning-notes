# 虚拟环境

## 1. 知识点定义

虚拟环境（Virtual Environment）是 Python 提供的一种**环境隔离机制**。它在一个独立的目录中创建一个"干净"的 Python 运行环境，包含独立的 Python 解释器和独立的第三方包安装目录。

你可以把虚拟环境理解为：给你的每个项目分配一个**专属的、隔离的 Python 小世界**，互不干扰。

为什么需要虚拟环境？想象以下场景：

- **项目 A** 需要 `Django==3.2`，**项目 B** 需要 `Django==5.0`。两个版本不兼容，安装在同一个 Python 环境中必然冲突。
- 你在电脑上尝试一个新库，结果安装失败，还把系统 Python 弄坏了。
- 你把代码发给同事，同事的 Python 环境里装了不同版本的依赖，你的代码跑不起来。

| 问题 | 没有虚拟环境 | 有虚拟环境 |
|------|-------------|-----------|
| 版本冲突 | 只能保留一个版本，项目间互相干扰 | 每个项目独立一套依赖 |
| 环境污染 | 全局安装的包越来越多，难以管理 | 每个环境精简，只含项目需要的包 |
| 系统安全 | `sudo pip install` 可能破坏系统工具 | 虚拟环境完全隔离，不影响系统 Python |
| 环境复现 | 难以精确知道项目依赖了哪些包 | `pip freeze` 导出精确依赖清单 |

> 小白理解：
> 虚拟环境就像给每个项目分配了一间独立的厨房。项目 A 用川菜调料，项目 B 用粤菜调料，互不串味。如果没有虚拟环境，所有项目的调料全混在一个厨房里，味道就全乱了。

## 2. 语法格式

### 2.1 创建虚拟环境

```bash
# 在项目根目录下创建名为 venv 的虚拟环境
python3 -m venv venv
```

这会在当前目录下创建一个 `venv/` 文件夹，其中包含：
- `bin/`（Windows 为 `Scripts/`）：Python 解释器、pip、激活脚本。
- `lib/`（Windows 为 `Lib/`）：安装的第三方包存放位置。
- `pyvenv.cfg`：配置文件。

> **命名习惯**：虚拟环境目录通常命名为 `venv`、`.venv` 或 `env`。推荐使用 `venv`。

### 2.2 激活虚拟环境

**macOS / Linux：**
```bash
source venv/bin/activate
```

**Windows（CMD）：**
```cmd
venv\Scripts\activate
```

**Windows（PowerShell）：**
```powershell
venv\Scripts\Activate.ps1
```

激活成功后，终端提示符会变为 `(venv) $`。

### 2.3 退出虚拟环境

```bash
deactivate
```

### 2.4 标准工作流程

```bash
# 1. 创建项目目录
mkdir my_project && cd my_project

# 2. 创建虚拟环境
python3 -m venv venv

# 3. 激活虚拟环境
source venv/bin/activate   # macOS/Linux

# 4. 安装项目依赖
pip install requests flask

# 5. 导出依赖清单
pip freeze > requirements.txt

# 6. 退出虚拟环境
deactivate
```

## 3. 重点说明

### 3.1 验证是否在虚拟环境中

```bash
# 查看当前使用的 Python 路径
which python3
# 输出应该指向 venv 目录下的 Python
# 例如：/你的项目路径/venv/bin/python3

# 确认 pip 也指向虚拟环境
which pip3
```

### 3.2 .gitignore 配置

虚拟环境目录**不应该**提交到 Git 仓库：

```
# Python 虚拟环境
venv/
.venv/
env/
.env/
```

### 3.3 现代工具简介：Poetry 与 PDM

| 工具 | 特点 |
|------|------|
| venv + pip | Python 内置，无需安装，够用且稳定 |
| Poetry | 依赖自动解析 + 锁定版本 + 统一配置（`pyproject.toml`） |
| PDM | 新一代工具，遵循 PEP 582，速度更快 |

> **建议**：初学者先掌握 `venv` + `pip`，这是基础且足够应对大多数场景。

## 4. Demo 样例

```python
# -*- coding: utf-8 -*-
"""
虚拟环境演示脚本
本脚本演示如何在 Python 代码中检测当前是否在虚拟环境中
运行方式：先激活一个虚拟环境，然后执行 python 本文件.py
"""

import sys
import os


def check_virtual_env():
    """检测当前是否在虚拟环境中运行"""
    print("=" * 55)
    print("虚拟环境检测")
    print("=" * 55)

    # 方法1：检查 sys.prefix 和 sys.base_prefix 是否不同
    # 在虚拟环境中，这两个值不同；在系统 Python 中，它们相同
    in_venv = sys.prefix != sys.base_prefix

    # 方法2：检查 VIRTUAL_ENV 环境变量
    venv_path = os.environ.get("VIRTUAL_ENV", None)

    print(f"Python 可执行文件: {sys.executable}")
    print(f"sys.prefix: {sys.prefix}")
    print(f"sys.base_prefix: {sys.base_prefix}")
    print(f"VIRTUAL_ENV 环境变量: {venv_path}")

    if in_venv:
        print("\n[状态] 你正在虚拟环境中运行 Python！")
        print(f"[虚拟环境路径] {venv_path}")
        print(f"[说明] 在此环境中安装的包不会影响系统 Python。")
    else:
        print("\n[状态] 你正在系统 Python 环境中运行。")
        print("[建议] 推荐创建虚拟环境来隔离项目依赖。")
        print("[创建命令] python3 -m venv venv")


def show_venv_workflow():
    """展示虚拟环境的标准工作流程（注释形式）"""
    print("\n" + "=" * 55)
    print("虚拟环境标准工作流程（请在终端中执行）")
    print("=" * 55)

    workflow = """
# --- 一、创建虚拟环境 ---
python3 -m venv venv

# --- 二、激活虚拟环境 ---
# macOS / Linux:
source venv/bin/activate
# Windows CMD:
venv\\Scripts\\activate

# --- 三、在虚拟环境中安装包 ---
pip install requests numpy

# --- 四、导出依赖清单 ---
pip freeze > requirements.txt

# --- 五、退出虚拟环境 ---
deactivate

# --- 六、在其他机器上重建环境 ---
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
"""
    print(workflow)


def show_tool_intro():
    """简介 Poetry 和 PDM"""
    print("=" * 55)
    print("现代包管理工具简介")
    print("=" * 55)
    print("建议学习路径：先精通 venv + pip → 再探索 Poetry")


if __name__ == "__main__":
    check_virtual_env()
    show_venv_workflow()
    show_tool_intro()
    print("=" * 55)
    print("提示：请在终端中创建并激活一个虚拟环境来体验！")
    print("=" * 55)
```

运行结果示例（在虚拟环境中）：

```
=======================================================
虚拟环境检测
=======================================================
Python 可执行文件: /Users/user/my_project/venv/bin/python3
sys.prefix: /Users/user/my_project/venv
sys.base_prefix: /usr/local/Cellar/python@3.12/3.12.3
VIRTUAL_ENV 环境变量: /Users/user/my_project/venv

[状态] 你正在虚拟环境中运行 Python！
[虚拟环境路径] /Users/user/my_project/venv
[说明] 在此环境中安装的包不会影响系统 Python。
```

## 5. 典型例题（附带答案）

### 例题 1

题目：

在你的练习目录中，创建一个名为 `my_venv_test` 的虚拟环境，激活它，然后使用 `which python3` 和 `pip list` 验证当前环境。

答案：

```bash
# 1. 创建虚拟环境
$ python3 -m venv my_venv_test

# 2. 激活虚拟环境
$ source my_venv_test/bin/activate

# 3. 验证
(my_venv_test) $ which python3
/Users/user/.../my_venv_test/bin/python3
# 路径在虚拟环境目录下，说明成功

(my_venv_test) $ pip list
Package    Version
---------- -------
pip        24.0
setuptools 69.0.3
# 只有基础包，说明是干净的环境
```

### 例题 2

题目：

小王在 VS Code 中打开了一个使用虚拟环境的项目。他在集成终端中执行 `pip install requests` 成功了，但在 `.py` 文件中写 `import requests` 时，IDE 却提示"未解析的导入"。请诊断可能的原因并给出解决方案。

答案：

VS Code 中选择的 Python 解释器不是虚拟环境中的 Python，而是系统 Python。

解决方案：
1. `Cmd/Ctrl+Shift+P` → `Python: Select Interpreter`。
2. 在列表中找到路径包含 `venv/bin/python3` 的选项，选择它。
3. 选择后，重启 VS Code 终端或新建一个终端窗口。

## 6. 易错点

### 易错点 1：忘记激活虚拟环境

错误现象：执行 `pip install xxx` 后包安装了，但在项目中 `import xxx` 报 `ModuleNotFoundError`。

错误原因：没有激活虚拟环境，包被安装到了系统 Python 或其他虚拟环境中。

正确做法：
1. 确认终端提示符前有 `(venv)` 标识。
2. 用 `which python3` 验证当前 Python 路径是否指向虚拟环境。
3. 打开新终端窗口时，需要重新激活虚拟环境。

### 易错点 2：虚拟环境目录被提交到 Git

错误现象：`git push` 时发现要上传几百 MB 的文件。

错误原因：虚拟环境目录（`venv/`）被误提交到了 Git 仓库。

正确做法：
1. 在项目根目录创建 `.gitignore` 文件，添加 `venv/`。
2. 如果已经提交了：`git rm -r --cached venv/`。

### 易错点 3：移动或重命名项目目录后虚拟环境失效

错误现象：把项目文件夹移到另一个位置，激活虚拟环境时报错。

错误原因：虚拟环境中的脚本记录了创建时的**绝对路径**。移动目录后路径对不上。

正确做法：删除 `venv/` 目录，在新位置重新创建虚拟环境并安装依赖。

### 易错点 4：用错了 Python 版本创建虚拟环境

错误现象：项目要求 Python 3.11+，但虚拟环境中运行的却是 Python 3.9。

错误原因：创建虚拟环境时，使用了系统默认的 `python3` 命令，但该命令指向旧版本。

正确做法：用完整路径指定 Python 版本创建虚拟环境：

```bash
/usr/local/bin/python3.12 -m venv venv
```

## 7. 课后练习（附带答案）

### 练习 1：虚拟环境和非虚拟环境的区别

题目：

分别在不激活和激活虚拟环境的状态下执行 `pip list`，对比两次输出的包数量和内容有何不同。

答案：

```bash
# 未激活虚拟环境（系统 Python）
$ pip3 list
# 会列出系统安装的所有包，可能有几十上百个

# 激活虚拟环境后
(venv) $ pip list
# 只列出了 pip 和 setuptools，非常干净

# 这说明：虚拟环境是完全隔离的，它有自己独立的包安装目录。
```

### 练习 2：完整的依赖管理流程

题目：

模拟一个完整的项目依赖管理流程：创建项目目录 → 创建虚拟环境 → 安装 `flask` → 导出 `requirements.txt` → 退出环境。最后查看 `requirements.txt` 文件的内容。

答案：

```bash
# 1. 创建项目目录
$ mkdir flask_demo && cd flask_demo

# 2. 创建+激活虚拟环境
$ python3 -m venv venv
$ source venv/bin/activate

# 3. 安装 flask
(venv) $ pip install flask

# 4. 导出依赖
(venv) $ pip freeze > requirements.txt

# 5. 退出
(venv) $ deactivate

# 6. 查看 requirements.txt
$ cat requirements.txt
flask==3.0.2
Jinja2==3.1.3
# ...（flask 的依赖包也被自动记录）
```

### 练习 3：.gitignore 配置

题目：

在你的项目根目录创建一个 `.gitignore` 文件，至少包含对虚拟环境目录和 `__pycache__` 目录的忽略规则。

答案：

```gitignore
# Python 虚拟环境
venv/
.venv/

# Python 缓存
__pycache__/
*.py[cod]

# IDE 配置
.vscode/
.idea/

# 系统文件

```

## 8. 答题总结

- 虚拟环境是 Python 的项目隔离机制，每个项目一个独立环境，互不干扰。
- 核心命令：`python3 -m venv venv`（创建）、`source venv/bin/activate`（激活）、`deactivate`（退出）。
- 虚拟环境目录**不要提交到 Git**，通过 `requirements.txt` 可完整重建。
- 每次开新终端都需要重新激活虚拟环境。
- 移动项目目录后，虚拟环境可能失效，需要删除后重建。
- IDE 中的 Python 解释器必须指向虚拟环境中的 Python（`venv/bin/python3`）。
- 学习路径：先精通 `venv` + `pip` → 再探索 Poetry/PDM 等现代工具。
